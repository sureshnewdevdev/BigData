Topic 1 Cloud Computing Definition

1) What is cloud computing in one sentence?
On‑demand delivery of shared IT resources and managed services over the internet with pay‑as‑you‑go pricing.
2) Name three essential characteristics defined by NIST.
On‑demand self‑service, broad network access, resource pooling, rapid elasticity, and measured service (any 3).
3) How is cloud different from traditional hosting?
Elastic, metered, API‑driven, multi‑tenant, with abstracted infrastructure vs fixed capacity, manual ticketing, and long procurement cycles.
4) Give two business benefits.
Faster time‑to‑market and lower upfront cost; plus global reach and improved availability.
5) What is resource pooling?
Provider’s resources are shared among tenants using a multi‑tenant model with dynamic allocation and isolation controls.
6) What is measured service?
Automatic metering of usage (compute hours, requests, GB‑months) to enable chargeback/showback and auto‑scaling policies.
7) What is an API in the cloud context?
A programmatic interface (REST/CLI/SDK) to provision and manage services, enabling IaC and automation.
8) Define tenancy.
The degree of isolation among customers sharing a platform, e.g., single‑tenant vs multi‑tenant deployments.
9) What is serverless in simple terms?
You deploy code, the platform manages servers and scaling, and you pay per execution/time.
10) Define cloud‑native.
Architecting apps with microservices, containers, DevOps/IaC, and managed services to exploit elasticity and resilience.
11) Example of cloud service for storage, compute, and database.
Object storage: S3/Blob/GCS; Compute: EC2/VM/Compute Engine; DB: RDS/SQL DB/Cloud SQL.
12) What is region?
A geographical area containing multiple isolated data‑center groups (AZs).
13) What is availability zone?
An isolated fault domain (one or more DCs) within a region with independent power/network.
14) Why is elasticity important?
It matches capacity to demand automatically, improving performance and reducing cost.
15) What is a managed service?
A service where the provider operates the underlying stack (patching, HA, backups) so customers focus on usage.
16) Role of IaC?
Infrastructure as Code (e.g., Terraform/Bicep/CloudFormation) declares resources so environments are reproducible and version‑controlled.
17) Define SLA.
Service Level Agreement—provider’s uptime/reliability commitment (e.g., 99.9%) and remedies.
18) What is cloud bursting?
A hybrid pattern where on‑prem handles baseline and cloud absorbs demand spikes.
19) What is data gravity?
Large datasets attract compute/services to where the data resides due to egress/latency—design compute near data.
20) Name two cloud adoption pitfalls.
Lift‑and‑shift without rightsizing and weak governance causing sprawl and cost overruns.

Topic 2 Cloud Computing Model Types (Deployment)

1) List deployment models.
Public, Private, Hybrid, Community, Multi‑cloud.
2) Public vs Private—key trade‑off?
Public: agility/cost; Private: control/compliance; Hybrid mixes both.
3) What is community cloud?
A cloud shared by organizations with common concerns (e.g., government, healthcare).
4) Typical hybrid connectivity?
VPN/ExpressRoute/Direct Connect + identity federation + routing with BGP.
5) Pattern: data residency with global app?
Keep data in‑region; serve via CDN and stateless edge/API Gateways; use multi‑region replication if needed.
6) Multi‑cloud anti‑pattern?
Lowest‑common‑denominator abstractions that reduce velocity and increase latency/egress cost.
7) When to choose private cloud?
Strict compliance, data sovereignty, or latency to on‑prem systems with existing DC investments.
8) Edge computing in models?
Push compute close to users/devices (CDN/edge runtimes) to reduce latency and bandwidth.
9) Control plane vs data plane?
Control plane manages resources (APIs, metadata); data plane handles workload/data traffic. Separation aids resilience.
10) Active‑active vs active‑passive regions?
Active‑active serves traffic in both regions; active‑passive fails over on disaster. Cost vs RTO/RPO trade‑off.
11) What is tenancy model choice impact?
Affects isolation, scaling, and data model—e.g., separate DB per tenant vs shared with tenant_id.
12) Network segmentation best practice?
Separate subnets per tier, use security groups/NSGs, private endpoints, and zero‑trust rules.
13) DR pattern for hybrid apps?
Replicate state to cloud, keep infra as code, periodically run failover drills (chaos testing).
14) How do SLAs compound across tiers?
Multiply availability of dependent services; design to avoid single points (e.g., decouple via queues).
15) What is landing zone?
A pre‑built, governed foundation (accounts/subscriptions, networking, IAM, policies) for workloads.
16) Cloud adoption framework purpose?
Guides strategy, governance, security, and platform setup to scale safely.
17) Blue/green vs canary in multi‑region?
Blue/green swaps entire stacks; canary shifts small traffic % to new version per region for safer rollout.
18) Typical network isolation for prod vs non‑prod?
Separate accounts/subscriptions/VPCs, distinct CIDRs, strict peering and role boundaries.
19) How to handle secrets across models?
Use managed secret vaults (KMS/Key Vault/Secret Manager), not env files; rotate and scope per environment.
20) Cost guardrails in landing zones?
Budgets/alerts, tagging policies, quotas, and automated cleanup of idle resources.

Topic 3 Cloud Computing Service Types

1) Core service categories?
Compute, Storage, Database, Networking, Security, Observability, AI/ML, Integration, Developer Tools.
2) Managed database advantages?
Automated backups, patching, HA, read replicas, point‑in‑time restore—reduced ops toil.
3) When to choose serverless compute?
Event‑driven, bursty workloads with sporadic traffic; pay per use and no idle cost.
4) Message queue vs pub/sub?
Queue: point‑to‑point with ordering. Pub/Sub: fan‑out to many subscribers; loosely coupled microservices.
5) CDN purpose?
Cache content at edge to reduce latency, offload origin, and cut egress from regions.
6) Observability pillars?
Logs, metrics, traces; add profiling and synthetics for completeness.
7) Storage classes typical choices?
Hot/standard for frequent access; cool/nearline for infrequent; archive for long‑term retention.
8) Data warehouse vs data lake?
Warehouse: structured, SQL, BI; Lake: raw multi‑format at low cost; Lakehouse blends both.
9) API gateways provide?
Routing, auth, rate limiting, transforms, observability for microservices.
10) Function timeout impact?
Long tasks should be offloaded to queues/workers; functions have limited runtime and memory.
11) Managed Kubernetes use case?
Portable microservices with custom runtimes/sidecars where PaaS isn’t flexible enough.
12) Event vs job schedulers?
Events react to triggers (e.g., blob upload); schedulers run time‑based CRON workloads.
13) DDoS/WAF services role?
Protect apps from volumetric and layer‑7 attacks with scrubbing, rules, and bot controls.
14) Secrets vs config?
Secrets are sensitive values stored in vaults; config are non‑secret settings managed via config stores.
15) Data integration services?
ETL/ELT tools like AWS Glue, Azure Data Factory, GCP Dataflow; orchestrate pipelines between sources and sinks.
16) AI/ML managed options?
SageMaker, Azure ML, Vertex AI for training/deployment; plus turnkey APIs for vision, speech, and generative AI.
17) Object lock/legal hold?
Write‑once‑read‑many (WORM) retention to prevent deletion/tamper—useful for compliance.
18) Private endpoints?
Expose PaaS services on private IPs inside VPC/VNet to avoid public internet paths.
19) Blueprints/templates?
Pre‑approved bundles of policies and resources to standardize environments.
20) Monitoring SLO vs SLA?
SLO is internal objective; SLA is contractual. Alerts should track SLO error budgets to act before SLA breaches.

Topic 4 IaaS, PaaS, SaaS (Comparisons & Scenarios)

1) Responsibility split across models?
SaaS: provider handles app‑to‑infra; PaaS: provider handles runtime/OS; IaaS: you handle OS up.
2) Migration of legacy app—start where?
Often IaaS lift‑and‑shift, then modernize to PaaS/serverless for ops/cost gains.
3) DB choices across models?
IaaS: self‑managed DB on VM; PaaS: managed DB (RDS/SQL DB/Cloud SQL); SaaS: embedded storage inside app (limited control).
4) Custom networking requirement?
Choose IaaS or container platforms; PaaS/SaaS expose limited networking customizations.
5) Compliance data—better fit?
PaaS with private endpoints and key management or Private Cloud IaaS; SaaS only if it meets certifications and data residency.
6) Cost model difference?
IaaS bills per VM/storage/network; PaaS per instance/runtime tier; SaaS per seat or usage (requests/records).
7) Lock‑in risk?
Highest in SaaS (proprietary features/data schema), moderate in PaaS, lowest in IaaS (but still with APIs).
8) Dev velocity?
PaaS/SaaS accelerate delivery via managed runtimes and integrations; IaaS requires more ops effort.
9) Examples of each?
IaaS: EC2/Azure VMs/GCE; PaaS: App Service/Elastic Beanstalk/Cloud Run; SaaS: Salesforce/Office 365/Slack.
10) Scaling approach difference?
IaaS: VMSS/ASG; PaaS: instance counts/tiers; SaaS: provider auto‑scales; you tune plan limits.
11) Backup strategy difference?
IaaS: you design/operate; PaaS: configure retention; SaaS: rely on vendor + export/archive.
12) Observability?
IaaS: full host metrics/logs; PaaS: app‑level logs/metrics; SaaS: limited telemetry via audit/API.
13) CI/CD fit?
Works everywhere; PaaS/Functions integrate tightly with build/deploy hooks.
14) When choose SaaS over PaaS?
For commodity capabilities (email, CRM, service desk) where customization needs are low.
15) Security patching impact?
IaaS: you patch OS; PaaS: provider patches runtime/OS; SaaS: provider patches app and stack.
16) Data portability concern?
Ensure export formats/APIs and retention windows before adopting SaaS/PaaS.
17) Cost predictability?
SaaS per‑user plans are predictable; IaaS/PaaS vary with workload patterns.
18) Performance tuning?
IaaS exposes most knobs (kernel/FS); PaaS exposes app/runtime options; SaaS exposes limited settings.
19) Example modernization path.
EC2 VM → container on EKS/AKS → PaaS App Service/Cloud Run → Functions for event flows.
20) FinOps implication across models?
Rightsize and schedule IaaS; choose proper tiers for PaaS; manage licenses/seat counts for SaaS.

Topic 5 Cloud Challenges

1) Top three risks?
Misconfiguration, IAM over‑permissioning, and uncontrolled cost.
2) How to prevent public bucket leaks?
Block public access, use bucket policies/ACL audits, enable object ownership, and scanners.
3) Identity sprawl mitigation?
Central IdP, SSO, role‑based access, short‑lived tokens, and periodic reviews.
4) Shadow IT control?
Guardrails with policies, budgets, and approved catalogs; network egress monitoring.
5) Data residency?
Choose compliant regions, encrypt with customer‑managed keys, and restrict cross‑region replication.
6) Vendor lock‑in?
Abstract at data/contracts, standardize with containers/IaC, and negotiate exit clauses.
7) Cost spikes?
Enable budgets/alerts, anomaly detection, and tagging; kill idle resources automatically.
8) Secrets leakage?
Use secret managers and workload identity; avoid hard‑coding/committing keys.
9) Multi‑AZ design mistake?
Putting stateful single‑AZ dependencies behind a multi‑AZ app—causes partial failures.
10) Noisy neighbor issue?
Use appropriate instance types, quotas, and dedicated hosts where needed.
11) Data egress surprise bills?
Architect compute near data, use CDNs and private peering, compress payloads.
12) Key rotation best practice?
Automate via KMS/Key Vault schedules, enforce least privilege, and monitor usage.
13) DR drills?
Run regular failover tests and chaos experiments to validate RTO/RPO.
14) Compliance mapping?
Map controls to frameworks (ISO, SOC, HIPAA) using provider attestations and shared responsibility.
15) Observability debt?
Define SLOs early, standardize logging/metrics schemas, and centralize traces.
16) Stateful microservices?
Externalize state to managed stores; use idempotency, retries, and sagas.
17) Capacity limits/quotas?
Request increases early for scale events; design sharding and region fallback.
18) IaC drift?
Block console changes, use drift detection and pipelines with policy checks.
19) Cost allocation to teams?
Use tags/labels & billing accounts; showback dashboards and chargeback.
20) Blue team basics in cloud?
Centralize logs, turn on threat detection (GuardDuty/Defender/SCC), least privilege, MFA, and just‑in‑time access.

Topic 6 Cloud Pricing & FinOps

1) Main cost drivers?
Compute hours, storage GB‑months, data transfer/egress, managed service units, and licenses.
2) On‑demand vs reserved?
On‑demand flexible; reserved/committed use saves 30–70% for steady workloads.
3) Spot/preemptible use case?
Fault‑tolerant, batch, CI, and stateless workers that can handle interruptions.
4) Right‑sizing?
Match instance size/tier to actual utilization via metrics; remove over‑provisioning.
5) Storage lifecycle policies?
Move cold data to cheaper classes automatically; apply deletion after retention.
6) Egress minimization?
Place compute near data, use CDNs/peering, and avoid chatty cross‑region calls.
7) License considerations?
BYOL vs included licenses (SQL, Windows); check core counts and multiplexing rules.
8) Scheduler savings?
Shut down dev/test after hours; scale to zero on serverless/container platforms.
9) Cost allocation?
Use tags/projects/resource groups per team/product; export billing to a data warehouse.
10) FinOps loop?
Inform (visibility), Optimize (rightsizing/commitments), and Operate (governance, budgets).
11) DB cost levers?
Choose right engine/tier, read replicas vs over‑sizing, storage autoscale, and backup retention.
12) Function pricing units?
Requests × (memory allocated × execution time) plus network/trigger costs.
13) Data warehouse pricing?
Provisioned capacity vs serverless; storage billed separately; consider result cache and clustering.
14) Logging costs?
Ingestion + retention; sample noisy logs, set shorter retention for debug environments, export to cold storage.
15) Savings plans vs RIs?
Savings plans are flexible across families; RIs are instance/region specific but can be cheaper in cases.
16) Cost of HA?
Multi‑AZ doubles some resources; weigh against RTO/SLO risk; use asynchronous read replicas when possible.
17) KPI for cost efficiency?
$/request, $/transaction, $/user, or cost‑to‑serve aligned with business metrics.
18) Budget enforcement?
Budgets with action policies, org policies to block expensive SKUs, and automated teardown.
19) Tag hygiene?
Mandatory keys like env, owner, cost_center, app; validate via policy and CI.
20) Quick wins for new tenants?
Turn on budget alerts, disable public IP by default, set lifecycle rules, and schedule dev shutdowns.

Topic 7 AWS / Azure / GCP

1) Core compute equivalents?
AWS EC2, Azure Virtual Machines, GCP Compute Engine.
2) Managed Kubernetes?
EKS (AWS), AKS (Azure), GKE (GCP).
3) Object storage?
S3, Azure Blob Storage, Google Cloud Storage.
4) Serverless functions?
Lambda, Azure Functions, Cloud Functions/Cloud Run Jobs.
5) Data warehouse?
Redshift (AWS), Azure Synapse/SQL DW, BigQuery (GCP).
6) IAM core concept?
AWS IAM users/roles/policies; Azure Entra ID + RBAC; GCP IAM with principals/roles/policies.
7) Networking primitives?
VPC (all three) with subnets, route tables; Security Groups (AWS/Azure NSGs), Firewalls (GCP).
8) DevOps/IaC tools?
CloudFormation/CDK, Bicep/ARM, Deployment Manager; multi‑cloud: Terraform/Pulumi.
9) Monitoring stacks?
CloudWatch, Azure Monitor (Log Analytics + App Insights), Cloud Monitoring/Logging.
10) Database managed services?
RDS/Aurora, Azure SQL/Database for PostgreSQL, Cloud SQL/Spanner/Firestore.
11) AI/ML platforms?
SageMaker, Azure ML, Vertex AI.
12) Queues and pub/sub?
SQS & SNS/EventBridge; Azure Storage Queues & Service Bus; Pub/Sub.
13) CDN offerings?
CloudFront, Azure Front Door/CDN, Cloud CDN.
14) Hybrid connectivity?
Direct Connect, ExpressRoute, Cloud Interconnect.
15) Cost tooling?
AWS Cost Explorer/Budgets, Azure Cost Management, GCP Billing reports/BUCKET exports.
16) Identity integration with Microsoft 365?
Tightest with Azure (Entra ID). AWS/GCP integrate via federation/SAML/OIDC.
17) Data analytics strengths?
AWS breadth + Glue; Azure Synapse + Fabric ecosystem; GCP BigQuery + Dataflow + strong network.
18) Free tiers?
All offer limited free usage; always check expiration (12‑month vs always‑free) and region limits.
19) Security centers?
AWS Security Hub/GuardDuty; Microsoft Defender for Cloud; Security Command Center (GCP).
20) Choosing a provider—quick rubric?
Match to skills, existing vendor deals, required services/regions, support model, and TCO including egress and ops.
