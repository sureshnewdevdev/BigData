A detailed guide covering fundamental concepts for organizing data for efficient analysis and reporting.

1. Modeling

Data Modeling is the process of creating a structured, visual representation of data and the relationships between different data elements. It ensures data is organized for optimal storage, retrieval, and analysis.

  • Goal: Define the data structure to meet business requirements and optimize database performance.
  • Output: A blueprint (model) used by developers and database administrators.

2. Conceptual/Logical/Physical Modeling

Data modeling progresses through three distinct stages of increasing detail:

2.1. Conceptual Data Model (CDM)

  • Focus: Business View (What). Defines major entities and the relationships between them.
  • Independence: Technology-agnostic (no data types, no tables).
  • Example: A CUSTOMER entity is linked to an ORDER entity by a "places" relationship.

2.2. Logical Data Model (LDM)

  • Focus: Structure (How). Details all attributes (columns) for each entity and defines primary and foreign keys.
  • Independence: Still independent of a specific database vendor (e.g., uses generic data types).
  • Example: The CUSTOMER entity has attributes: Customer_ID (PK), First_Name, Email.

2.3. Physical Data Model (PDM)

  • Focus: Implementation (How to Store). Defines the actual database objects, including tables, columns with specific data types (e.g., VARCHAR, INT), indexes, and constraints specific to the chosen database (e.g., MySQL, BigQuery).
  • Example: Defining the CUSTOMER table with Customer_ID as INT64 and applying an index on the Email column.

3. Dimension Modeling

A technique pioneered by Ralph Kimball, optimized for data warehousing and business intelligence (BI). It separates quantitative metrics from descriptive context.

ComponentPurposeExample Attributes
Fact TableStores measurable business events (metrics or KPIs).Sales_Amount, Quantity_Sold, Foreign Keys to all Dimensions.
Dimension TableStores descriptive context (who, what, where, when, how).Product_Name, Store_City, Date, Customer_Age.

4. Star Schema ‚≠ê

The simplest and most efficient Data Warehouse schema. It is called a "star" because the design resembles a star shape.

  • Structure: A single, central Fact Table is directly connected to multiple Dimension Tables.
  • Relationship: Dimensions are typically de-normalized (no further joins required).
  • Advantage: Very fast query performance due to minimal joins.

Diagram Concept

[Date Dim] <--> [Sales Fact] <--> [Product Dim]
[Customer Dim] <--> [Sales Fact] <--> [Store Dim]

5. Snowflake Schema ‚ùÑÔ∏è

A logical extension of the Star Schema where the dimension tables are normalized to reduce redundancy.

  • Structure: Dimension tables are broken down into sub-dimensions (hierarchies).
  • Example: The Product dimension is normalized into separate Product and Product Category tables.
  • Disadvantage: More complex queries (more joins required), which can slow down reporting compared to the Star Schema.

Diagram Concept

[Date Dim] <--> [Sales Fact] <--> [Product Dim] <--> [Product Category]

6. Slowly Changing Dimensions (SCD)

Techniques used to handle dimension attributes that change over time while maintaining an accurate historical record for reporting.

  • Problem: If a customer's address changes, how do we report on orders shipped to the old address vs. the new address?
  • Solution: Use different SCD types to manage the historical context.

7. SCD Types

7.1. SCD Type 1: Overwrite (No History)

  • Method: The new value replaces (overwrites) the old value.
  • History: Lost. All past fact records will appear to use the new value.
  • Use Case: Correcting a mistake (typo) or changing a non-critical attribute (e.g., customer's email).

7.2. SCD Type 2: Add New Row (Full History)

  • Method: A new row is created for the new attribute value. The dimension table uses columns like Start_Date, End_Date, and an Is_Current flag.
  • History: Fully preserved. Fact records link to the row that was active at the time of the transaction.
  • Use Case: Tracking critical changes like **customer address** or **employee department**. This is the most common SCD type.
Before Change:
| Cust_Key | Name | Address | Is_Current | Start_Date | End_Date |
| 101 | Alice | Old Street | Y | 2024-01-01 | NULL |

After Change (New Row Added):
| 101 | Alice | Old Street | N | 2024-01-01 | 2024-06-30 |
| 102 | Alice | New Street | Y | 2024-07-01 | NULL |

7.3. SCD Type 3: Add New Column (Limited History)

  • Method: A new column is added to the dimension table to store the previous value.
  • History: Limited to tracking only the current value and the **immediate previous value**.
  • Use Case: Tracking attributes where you only need to compare the current state to the last state (e.g., a customer's loyalty tier).
